# Алгоритм reconciliation

> **Примечание:** Это описание проверенного в реализации варианта алгоритма. Он работает и протестирован, но это не единственный возможный подход. Можно придумать другую архитектуру (например, fibers как в React, или полностью итеративный обход) — главное чтобы поведение для пользователя совпадало со SPEC.

## Основная идея

Однопроходный алгоритм: обходим новое дерево (результат render), для каждого узла ищем старый элемент по идентификатору (key или позиция), переиспользуем или создаём новый. Возвращаем список DOM-узлов в правильном порядке.

В отличие от классического двухпроходного подхода (отдельный reconcile + отдельная синхронизация DOM) — всё за один проход.

## Идентификаторы элементов

Для каждого узла строится **идентификатор** — ключ, по которому старый элемент ищется в карте.

- Если у элемента есть `props.key` → идентификатор = `#` + key
- Если key нет → идентификатор = путь (например, `,0,1,2` — позиция в дереве)

Экранирование:
- Запятая в key удваивается: `,` → `,,` (чтобы key с запятой не ломал парсинг пути)
- `#` в начале — зарезервированный маркер «это user key»

Это даёт два свойства:
- **Элементы с key** могут перемещаться между позициями — идентификатор `#key` не зависит от того, где элемент находится
- **Элементы без key** переиспользуются только на той же позиции — их идентификатор привязан к пути

### Наследование идентификатора детьми

Дети элемента с key наследуют `#key` как префикс своих идентификаторов.

Пример — vnode с key='fio':
```
h(X, { key: 'fio' },        // #fio
    h('E'),                  // #fio,0   ← наследует #fio
    h('Y')                   // #fio,1
)
```

Поэтому когда keyed-элемент перемещается между родителями, его unkeyed-дети тоже переиспользуются — их идентификаторы `#fio,0`, `#fio,1` не меняются при перемещении.

Это же правило работает для keyed Fragment: дети получают идентификаторы `#fragmentKey,0`, `#fragmentKey,1`.

### Duplicate keys

User keys должны быть уникальны в пределах одного render. Дубликаты — ошибка разработчика, но алгоритм обрабатывает их детерминированно:

- **Первый** элемент с данным key считается правильным, получает идентификатор `#key`
- **Последующие** элементы с тем же key — дубликаты. Считаются как элементы без key, получают path-based идентификатор
- В development mode выводится предупреждение
- В production mode предупреждение не выводится, но поведение то же

## Карта старых элементов (keyMap)

**Персистентная** карта на instance (компонента) / на container (mount). Содержит актуальную карту текущего дерева.

- На первом render — пустая
- Во время reconcile алгоритм ищет в ней старые элементы по идентификатору (`keyMap.get(uid)`)
- И **записывает** новые vnode в неё (`keyMap.set(uid, newVnode)`) — для следующего render
- После reconcile — cleanup: для ключей, которых нет в новом дереве → размонтировать старый элемент и **удалить из keyMap** (`keyMap.delete(key)`)

**Важно:** `populateKeyMap` (перестроение keyMap из старого vdom перед reconcile) **не нужен** — keyMap всегда актуален после прошлого reconcile + cleanup.

**Важно:** Каждая функция-обработчик (HTML, Component, Fragment, Portal) **обязана** делать `keyMap.set` для своего vnode. Если забыть — элемент не найдётся на следующем render, будет пересоздаваться.

### Cleanup

После основного прохода:
- Проходим по keyMap
- Для каждого идентификатора, которого **нет** в Set использованных → размонтируем старый элемент и удаляем из keyMap
- Размонтирование удаляет DOM-узел и рекурсивно вызывает `onUnmounted` для вложенных компонентов, `ref(null)` и т.д.

На первом render (старого дерева нет) — cleanup не нужен, keyMap пустой.

## Set использованных идентификаторов (actualUIDs)

Set для двух целей:
1. **Duplicate detection** — если идентификатор уже в Set, это duplicate key
2. **Cleanup** — ключи в keyMap, которых нет в actualUIDs → удалять

В production mode на первом render (когда старого дерева нет) — можно использовать заглушку вместо Set (дешевле, cleanup не нужен, duplicate detection пропускается). В development mode — настоящий Set (для предупреждений о дубликатах).

## Основной проход

Для каждого узла нового дерева:

1. **Массив:**
   - Для каждого элемента — рекурсивно обработать, результаты добавить в общий массив
   - `null`/`false` в массиве пропускаются (не создают DOM-узел)

2. **Текстовый узел:**
   - Ищет в keyMap по идентификатору
   - Если нашёл текстовый → переиспользует, обновляет текст если изменился
   - Если нашёл нетекстовый → размонтирует старый, создаёт новый текстовый узел
   - Если не нашёл → создаёт новый

3. **HTML элемент:**
   - Ищет в keyMap по идентификатору
   - Если нашёл с тем же тегом:
     - Переиспользует DOM
     - Проверяет, не нужно ли пересоздать (атрибуты `type`, `is`, `multiple` для select — нельзя поменять «на лету»)
     - Если пересоздание не нужно → shallow compare props, применяет изменившиеся
   - Если не нашёл или тег не совпадает:
     - Создаёт новый DOM
     - Размонтирует старый элемент (если был)
   - **Записывает vnode в keyMap** (обязательно!)
   - Рекурсивно обрабатывает детей
   - Выстраивает детей в правильном порядке (см. ниже)
   - Особый случай `SELECT`: значение `value` применяется **после** выстраивания детей — нужно чтобы `<option>` уже были в DOM для установки `selected`

4. **Компонент:**
   - Ищет в keyMap по идентификатору
   - Если нашёл с тем же типом (та же фабрика):
     - Переиспользует instance
     - Обновляет входящие props, контекст, namespace
     - Запускает внутренний rerender (props → memo → возможно render)
     - Возвращает DOM-узлы instance (без выстраивания — это делает родитель)
   - Если не нашёл или тип не совпадает:
     - Создаёт новый instance
     - Навешивает API (update, context, refs, rerender)
     - Вызывает init (если есть)
     - Запускает первый render
     - Вызывает ref (если есть)
     - Добавляет instance в очередь для onMounted
   - **Записывает vnode в keyMap** (обязательно!)

5. **Fragment:**
   - Если есть key и найден старый Fragment → переиспользует
   - Иначе → создаёт новый
   - Рекурсивно обрабатывает детей, используя свой идентификатор как префикс (для наследования)
   - **Записывает vnode в keyMap**

6. **Portal:**
   - Переиспользует instance если есть, иначе создаёт (с anchor-узлом)
   - Обрабатывает детей через отдельную логику с тремя случаями:
     - Контейнер появился → первый mount детей
     - Контейнер исчез → unmount детей
     - Контейнер есть → reconcile (с перемещением узлов если контейнер сменился)
   - Возвращает `[anchor]` — anchor это пустой текстовый узел, который держит место портала в родительском DOM
   - **Дети портала используют родительский keyMap** (portal не изолирует детей — он как компонент без своих элементов, только children)

## Выстраивание детей в DOM

После обработки всех детей HTML-элемента (или корня mount) DOM-узлы нужно выстроить в правильном порядке.

Алгоритм:
1. **Быстрая проверка:** если старый и новый списки идентичны (та же длина + те же узлы по индексам) → ничего не делаем
2. **Перемещение:** выстраивает новые узлы в нужном порядке через `insertBefore`. Идёт с конца, чтобы не ломать `nextSibling`.

**Важно:** этот шаг **не удаляет** старые узлы, которых нет в новом списке. Удалением занимается отдельный cleanup — потому что при удалении нужно вызвать `onUnmounted`, `ref(null)` и другие lifecycle-хуки, а не просто `removeChild`.

## ref (проброска мостов)

`ref` — это проброска ссылки на DOM-узел или instance во внешний код. Не предназначен для побочных эффект (focus, scroll и т.д.) — для этого есть lifecycle hooks.

Поэтому ref вызывается **инлайн** в момент создания/переиспользования узла, прямо в функции-обработчике:
- HTML: `ref(dom)` после создания/переиспользования dom
- Component: `ref(inst)` после rerender
- Portal: `ref(inst)` после создания instance
- При unmount: `ref(null)` (в функции размонтирования)

Отдельный обход дерева для ref'ов **не нужен**.

## onMounted (lifecycle)

`onMounted` должен вызываться **children-first** (дети раньше родителей) и только при первом mount компонента.

Алгоритм:
- Во время reconcile при создании нового компонента — добавляем instance в **очередь** (массив)
- После всего reconcile — вызываем `onMounted` для каждого в **обратном порядке** (последний добавлен = самый глубокий ребёнок = вызывается первым)
- Отдельный обход дерева **не нужен**

Очередь хранится на instance (для вложенных rerender'ов — своя очередь, с сохранением родительской).

## Где вызывается алгоритм

1. **Поверхностный mount** — точка входа пользователя. При повторном mount того же контейнера:
   - keyMap персистентный (хранится рядом с container)
   - Основной проход → cleanup → выстраивание детей в корне

2. **Внутренний rerender компонента** — когда компонент обновляется:
   - keyMap персистентный на instance
   - Основной проход → cleanup
   - Выстраивание детей делает **родитель** компонента (через HTML-обработчик или mount)

3. **Memo-skip path** — когда memo заблокировала render:
   - Алгоритм не запускается, вместо этого рекурсивно вызывается rerender у дочерних компонентов
   - Контекст протекает через memo-защищённые компоненты (это их свойство)

## Ключевые отличия от классического подхода

| Классический подход | Этот алгоритм |
|---------------------|---------------|
| Двухпроходный (reconcile + sync) | Однопроходный |
| Сопоставление по позиции | Сопоставление по key/пути |
| Отдельный обработчик для массивов | Один обработчик для всех узлов |
| HTML-элементы не выстраивают детей | HTML-элементы сами выстраивают детей |
| Компоненты сами синхронизируют себя | Компоненты НЕ синхронизируют себя (делает родитель) |
| keyMap перестраивается каждый раз | keyMap персистентный, инкрементальный |
| ref через отдельный обход | ref инлайн при создании |
| onMounted через отдельный обход | onMounted через очередь, собираемую при создании |

## Пример

**Старое дерево:**
```html
<div>
  <span key="a">A</span>
  <span key="b">B</span>
  <span key="c">C</span>
</div>
```

**Новое дерево:**
```html
<div>
  <span key="c">C</span>
  <span key="a">A</span>
  <span key="b">B</span>
</div>
```

**Процесс:**
1. keyMap (от прошлого render + cleanup) содержит: `{'#a': spanA, '#b': spanB, '#c': spanC}`
2. Основной проход по новому дереву:
   - `<span key="c">` → находит spanC в keyMap → переиспользовать → `keyMap.set('#c', newSpanC)`
   - `<span key="a">` → находит spanA в keyMap → переиспользовать → `keyMap.set('#a', newSpanA)`
   - `<span key="b">` → находит spanB в keyMap → переиспользовать → `keyMap.set('#b', newSpanB)`
3. Выстраивание детей перемещает узлы в порядок [C, A, B]
4. DOM-узлы переиспользованы, порядок исправлен

## Производительность

**Плюсы:**
- Один проход вместо двух
- Быстрая проверка порядка при выстраивании (O(n) с ранним выходом)
- Переиспользование элементов без key на той же позиции
- Переиспользование детей keyed-элементов при перемещении родителя (через наследование идентификаторов)
- Инкрементальный keyMap — не перестраивается каждый раз
- ref и onMounted без отдельных обходов

**Минусы:**
- keyMap хранит все элементы (O(n) памяти)
- actualUIDs Set создаётся на каждый render (O(n) памяти, кроме заглушки в prod на первом mount)
- На update без memo — полный O(n) обход, нет bailout по неизменённым поддеревьям

## Возможные альтернативы (для других реализаций)

1. **Fibers (как React)** — связный список child/sibling/return, итеративный обход, bailout по shallow compare. Лучше для update-сценариев без memo, но сложнее.
2. **Полностью итеративный обход со стеком** — пробовали, оказался хуже рекурсии (аллокация task объектов, V8 оптимизирует рекурсию лучше).
3. **LIS (Longest Increasing Subsequence)** для минимального количества перемещений в syncDOMChildren — для списков с большим количеством перестановок.
4. **Кэширование DOM-узлов старого дерева** — сейчас при повторном mount делается лишний O(n) проход для сбора DOM-узлов. Можно сохранять их при первом mount.
