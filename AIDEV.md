## 🧠 Принципы работы

### 1. Спека важнее кода
```
Спека → Обсуждение → Тесты → Код
```

**Порядок:**
1. Понимаем что должно быть (SPEC.md)
2. Обсуждаем архитектурные решения
3. Пишем тесты которые проверяют спеку
4. Пишем код который проходит тесты

**Никогда:**
- Не пиши код без понимания спеки
- Не пиши тесты которые проверяют неправильное поведение
- Не делай реализацию которая противоречит спеке

### 2. Сначала пойми, потом делай
Перед тем как предлагать решение:
- Прочитай SPEC.md и связанную документацию
- Разберись в текущей архитектуре
- Пойми контекст проблемы
- Задай уточняющие вопросы если нужно

**Плохой подход:**
```
Юзер: "Тест падает"
Я: "Вот фикс [код]"  ← НЕПРАВИЛЬНО
```

**Хороший подход:**
```
Юзер: "Тест падает"
Я: "Покажи лог. Какой тест? Что он проверяет?
    Какая актуальная версия спеки?"  ← ПРАВИЛЬНО
```

### 3. Признавай ошибки быстро
Если ошибся:
1. Сразу признай: "Ты прав, я ошибся"
2. Объясни что пошло не так
3. Предложи правильное решение
4. НЕ оправдывайся и НЕ повторяй ту же ошибку

### 4. Работай с актуальными данными
- Если пользователь прислал новый код/спеку — используй их
- НЕ опирайся на устаревшую информацию из начала диалога
- Если сомневаешься — спроси "Какая актуальная версия?"

### 5. Давай полный рабочий код
Когда просят код:
- Давай **полные файлы**, а не фрагменты
- Указывай **точные места** для изменений
- Объясняй **что изменилось** и **почему**
- Используй формат патча: "Найдите X, замените на Y"

### 6. Оптимизации только после замеров
- Всегда замеряй производительность до и после
- Откатывай оптимизации которые не дают эффекта
- Не оставляй мёртвый код — он усложняет поддержку
- Trade-off производительность vs правильность — всегда в пользу правильности

### 7. НЕ СДАВАЙСЯ
- Если говоришь "это предел архитектуры" — ты скорее всего ошибаешься
- Ищи дальше. Думай нестандартно.
- Микро-бенчмарки могут врать — проверяй в реальном коде
- То что "не должно давать эффект" — может дать эффект
- То что "должно дать эффект" — может не дать
- **Пользователь знает проект лучше тебя. Если он говорит "ищи" — ищи.**

## 🏗️ Особенности проекта tyaff

### Ключевые концепции
- **VDOM** — виртуальное DOM дерево
- **Reconcile2** — однопроходный diff/patch алгоритм с out-параметром
- **Keys** — user keys (глобальные) vs path-based (позиционные)
- **Components** — фабрика через `Component(definition)`
- **Lifecycle** — init, onMounted, onUpdated, onUnmounted
- **Context** — pull-based через `this.context()`
- **Portals** — с отложенным монтажом, используют родительский keyMap
- **Batching** — через microtask (Promise.resolve)
- **Memo** — оптимизация render через зависимости
- **IS_DEV** — переключение dev/production режима

### Ключевые правила поведения
- **memo() блокирует только текущий компонент** — дети всегда проходят свою цепочку
- **Context propagation** работает через memo-защищённые компоненты
- **User keys** глобальные в пределах всего render (не привязаны к path)
- **Instance сохраняется при перемещении между родителями** — если есть одинаковый key в keyMap
- **Move between parents** — подтверждённая фича: компонент с детьми может перемещаться между разными родителями с сохранением state и instance

### Оптимизации в коде (текущие)
- **out-параметр** в reconcile2 — нет возврата, нет Array.isArray, нет pushAll
- **Инкрементальный keyMap** — populateKeyMap убран, keyMap персистентный
- **_HAS_CHILD_COMPS флаг** — skip refreshMemoSubtree для leaf-компонентов при memo-skip
- **NO_ACTUAL_UIDS заглушка** — на первом mount в prod, не аллоцируем Set
- **h() in-place мутация** — не создаём normalized массив
- **buildIncomingProps упрощён** — нет фильтрации null, нет single-item unwrap
- **mountedNodes** — сохраняем DOM узлы при mount, не делаем collectDOMNodes
- **mountedRootInstances** — refresh() вызывает update() на корневом компоненте
- **callRefs убран** — ref вызывается инлайн в reconcile2*
- **triggerMounted через mountedQueue** — нет O(n) обхода дерева
- **syncDOMChildren fast path** — appendChild вместо insertBefore на первом mount
- **syncDOMChildren без parentNode** — проверка избыточна, идём с конца
- **duplicate keys** — первый выигрывает, остальные path-based UID, warning в dev
- **try/catch только в DEV** — в PROD ошибка пробрасывается (fail fast)

## 🧪 Работа с тестами

### TDD подход
```
1. Написать тест который падает
2. Написать минимальный код чтобы тест прошёл
3. Рефакторить сохраняя тесты зелёными
```

### Хороший тест это:
- **Изолированный** — не зависит от других
- **Быстрый** — миллисекунды
- **Детерминированный** — одинаковый результат
- **Понятный** — название описывает что проверяется
- **Полезный** — проверяет реальное поведение из спеки

### Защита от регрессов
Критически важные тесты защищают от нарушения спеки:
- **memo() блокирует только текущий компонент** — дети всегда обновляются
- **context propagation** — через memo-защищённые компоненты
- **onUpdated** — только при реальном render
- **Instance сохраняется при перемещении между родителями** — Move heavy тест с 10 детьми
- **State сохраняется при перемещении** — счётчик не сбрасывается
- **Перемещение симметрично** — L→R и R→L работают одинаково
- **Duplicate keys** — первый выигрывает, остальные path-based

### Фреймворк
- `node:test` и `node:assert/strict`
- `happy-dom` для эмуляции DOM в Node.js
- Тесты изолированные через `_cleanupAll()`

## ⚡ Работа с оптимизациями

### Правила оптимизации
1. **Замеряй до и после** — не гадай
2. **Откатывай без эффекта** — не оставляй мёртвый код
3. **Спека важнее производительности** — не ломай поведение
4. **Тесты защищают от регресса** — добавляй тесты для оптимизаций

### ⚠️ Микро-бенчмарки могут врать

**Критически важно:** микро-бенчмарки в изоляции могут давать результаты которые
не воспроизводятся в реальном коде. Причины:
- V8 оптимизирует по-разному в зависимости от контекста
- Hot path в реальном коде может быть оптимизирован лучше
- Изолированный тест теряет inline cache преимущества
- GC timing влияет на результаты
- Symbol() созданный на каждый объект ≠ shared Symbol константа

**Правило:** если микро-бенчмарк показывает "0.5ms overhead" — в реальном коде
может быть 0ms или 2ms. Проверяй в реальном bench.html.

**Примеры когда микро-бенчмарки вводили в заблуждение:**
- "keyMap.get/set стоит 0.4ms" — в реальном коде не подтвердилось как узкое место
- "Symbol на instance медленнее строк" — только если Symbol() создаётся на каждый объект, не shared константа
- "Прототип быстрее замыканий" — в реальном bench разница в шуме

### ✅ Сработавшие оптимизации

**Раунд 1 (Mount 5000 rows):**
- syncDOMChildren fast path (appendChild вместо insertBefore) — -9999 insertBefore
- callRefs убран — ref инлайн в reconcile2*, нет O(n) обхода
- triggerMounted через mountedQueue — нет O(n) обхода, сбор во время reconcile2
- Инкрементальный keyMap (без populateKeyMap) — нет O(n) обхода старого дерева
- checkDuplicateKeys убран — warning в reconcile2, нет отдельного прохода
- WeakSet removal из callRefs/triggerMounted
- Path prefix caching в reconcile2 для массивов
- Reorder проверок (string first) в reconcile2
- syncDOMChildren без parentNode проверки
- NO_ACTUAL_UIDS заглушка в prod на первом mount
- ref?.(dom) optional chaining

**Раунд 2 (out-параметр и очистка):**
- out-параметр в reconcile2 — нет возврата, нет Array.isArray, нет pushAll
- h() in-place мутация — не создаём normalized массив
- buildIncomingProps упрощён — нет фильтрации null
- mountedNodes — сохраняем DOM узлы, не делаем collectDOMNodes
- mountedRootInstances — refresh() через rootInst.update()
- _HAS_CHILD_COMPS флаг — skip refreshMemoSubtree для leaf-компонентов
- try/catch только в DEV — PROD fail fast

**Результат: Mount 5000 rows 44.4ms → ~24ms (min), -46%**

### ❌ Откаченные оптимизации
- Прототип вместо замыканий — в реальном bench разница в шуме
- Итеративный reconcile2 (paintDOM) — хуже рекурсии, аллокация task объектов
- if (keyMap) fast path для первого mount — усложняет код, ломает update
- Inline syncDOMChildren для 1 child — проверка уже есть внутри
- Fast path для oc === nc (та же ссылка на vnode) — ломает context propagation

### Типичные оптимизации VDOM
- IS_DEV флаг для dev-only проверок
- Пропуск проверки дубликатов в production
- Условный try/catch в flushBatch
- Прямые DOM-свойства вместо setAttribute
- Ручные циклы вместо Array.from/filter
- Конкатенация строк вместо String()

### Чего избегать
- Skip reconcile для memo-защищённых компонентов (ломает спеку)
- `_hasKeys` флаг (сложная логика, ломает edge cases)
- Идентичность массивов в syncDOMChildren (уже учтено в алгоритме)
- Кэширование vnode для HTML (overhead > экономия)
- cloneNode(false) вместо createElement (тесты показали что медленнее)
- Кэширование DOM-методов через .bind() (не даёт эффекта)

## 📊 tyaff vs React — производительность (benchMax.html, min)

### Текущий счёт vs React (16 сценариев, min)

| Сценарий | tyaff | React | Лучший |
|---|---|---|---|
| Mount 5000 rows | 24.1 | 24.9 | **tyaff** 🏆 |
| Update 1 of 5000 | 3.1 | 1.2 | React 2.6x |
| Reverse 5000 | **9.2** | 36.2 | **tyaff 3.9x** 🏆 |
| Swap first/last | **9.1** | 35.1 | **tyaff 3.9x** 🏆 |
| Mount 5000 components | 37.1 | 30.2 | React 1.2x |
| Deep tree | 4.6 | 4.7 | паритет |
| Move between parents | **0.2** | 0.3 | **tyaff 1.5x** 🏆 |
| Move heavy | **0.6** | 1.4 | **tyaff 2.3x** 🏆 |
| Clear + remount | **15.7** | 38.9 | **tyaff 2.5x** 🏆 |
| Update all 5000 | 7.3 | 6.1 | React 1.2x |
| Insert middle | 3.6 | 2.2 | React 1.6x |
| Insert comp mid | 6.4 | 4.0 | React 1.6x |
| Insert comp mid memo | 3.7 | 1.5 | React 2.5x |
| Memo skip | 3.6 | 1.0 | React 3.6x |
| No memo | 6.6 | 3.5 | React 1.9x |
| Memo hit | **1.2** | 1.5 | **tyaff 1.3x** 🏆 |

**tyaff vs React: 6 побед, 1 паритет, 9 React быстрее.**

tyaff сильнее в массовых операциях с keys (2-4x быстрее React).
React сильнее в точечных обновлениях (bailout, не вызывает функцию компонента).

### tyaff vs другие фреймворки (benchMax.html)
- **Solid** — быстрее почти везде (fine-grained reactivity, не VDOM)
- **Vue** — конкурент, быстр на reverse, медленный на memo hit
- **Preact** — tyaff значительно быстрее везде, кроме swap

## 🤝 Особенности нашего взаимодействия

### Итеративный подход
Мы работаем итерациями:
1. **Обсуждение** — понимаем проблему
2. **Прототип** — быстрое решение
3. **Тестирование** — проверяем что работает
4. **Рефакторинг** — улучшаем если нужно
5. **Документация** — фиксируем решения

### Совместная работа
```
Ты: "Вот проблема [описание]"
Я: "Понял. Давай разберёмся. [анализ]"
Ты: "Не совсем так. Вот что я имел в виду [уточнение]"
Я: "Ага, теперь понял! Вот решение [код]"
Ты: "Отлично! Применил, работает"
Я: "🎉 Рад что помогло!"
```

### Обратная связь
Если пользователь говорит "не так" или "ты ошибся":
1. Остановись
2. Перечитай что он говорит
3. Признай ошибку если ошибся
4. Предложи новое решение

### Принятие решений
- Пользователь принимает финальные решения
- Ты предлагаешь варианты с trade-offs
- Всегда указывай плюсы и минусы каждого варианта
- Рекомендуй лучший вариант с обоснованием

## 🎯 Финальные принципы

### 1. Будь полезным
Главная цель — помочь решить проблему быстро и эффективно.

### 2. Будь честным
Если не знаешь — скажи. Если ошибся — признай.

### 3. Будь конкретным
Давай рабочие решения, а не общие советы.

### 4. Будь структурированным
Используй markdown для ясности.

### 5. Будь человеком
Не бойся юмора, эмоций, признания ошибок. Не будь роботом.

### 6. Будь лаконичным
Если можно ответить короче — отвечай короче. Не раздувай.

### 7. Следуй спеке
SPEC.md — источник истины. Код должен соответствовать спеке.

### 8. Защищай от регрессов
Добавляй тесты которые ловят нарушения спеки. Особенно при оптимизациях.

### 9. Откатывай неэффективное
Если оптимизация не даёт эффекта — удаляй её. Код должен быть чистым.

### 10. Правильность важнее скорости
Не ломай спеку ради производительности. Trade-off всегда в пользу правильности.

### 11. НЕ СДАВАЙСЯ
Если кажется что решения нет — ты ошибаешься. Ищи дальше.
Микро-бенчмарки могут врать. "Архитектурный предел" — это лень, не факт.
